## 为什么要自己搭一个博客

我一直觉得，个人博客真正有价值的地方，不只是把做完的项目展示出来，而是把一个人如何思考、如何踩坑、如何修正路线的过程留下来。

以前我更容易把作品当成一次性成果：项目写完，截图放一放，简历里加一句，然后继续下一个任务。但这次我想换一种方式，把自己的学习、开发、部署和复盘都整理进同一个系统里。

所以 CodePulse Blog 的目标很明确：它不是一个单纯的静态作品集，而是一个可以长期维护的技术成长记录站。

## 第一版先做结构，不急着堆内容

刚开始搭建时，我没有先追求复杂功能，而是先把信息架构想清楚。这个站点目前分成几个核心部分：首页、正在开发、项目库、工具箱、个人履历和文章详情页。

首页负责给出整体状态，让访客快速知道我是谁、最近在做什么、哪些项目值得看。正在开发和项目库用来区分项目生命周期：一个记录还在推进中的东西，一个沉淀已经相对稳定的成果。

这一步很重要。因为博客一旦开始写内容，如果结构本身不稳定，后面就会出现很多尴尬的问题：文章不知道归到哪里，项目更新只能靠手动改页面，内容越写越散。

## 从静态页面升级到内容系统

第一阶段最关键的变化，是把博客从静态页面升级成一个可维护的内容系统。

我选择了比较轻量的方案：内容暂时不接数据库，而是放在 `data` 目录下的 JSON 文件里。站点设置、项目列表、文章列表都走同一套文件型内容库，前台由 Astro 服务端读取，后台通过 API 写入。

这个方案不华丽，但非常适合当前阶段。对个人博客来说，真正重要的是内容能不能稳定写、稳定备份、稳定部署。数据库当然更强，但也会带来迁移、权限、备份和运维复杂度。现在用 JSON 反而让我可以把注意力放回内容本身。

后台也在这一阶段补齐了基本能力：登录、维护站点信息、编辑项目、编辑文章。虽然编辑体验还不算豪华，但已经足够支撑我开始正式写博客。

## 云端部署比本地跑通更真实

本地能跑起来只能说明代码没太大问题，云端能稳定访问才算真正进入可使用状态。

部署时我把 Astro 项目构建为 Node 服务，通过环境变量管理后台密码和会话密钥，再用 Nginx 做反向代理。这个过程也让我重新检查了一遍项目边界：哪些内容应该进仓库，哪些配置应该放 `.env`，哪些目录需要备份，哪些东西不能只靠本地习惯。

这一步带来的感觉很不一样。以前项目停在本地时，它更像一个练习；部署到云服务器之后，它就开始像一个需要被维护的产品。哪怕访问量暂时很小，我也需要认真对待稳定性、安全性和后续迭代。

## 当前版本的取舍

这个版本没有追求一步到位。

我没有直接上复杂 CMS，也没有做富文本编辑器，更没有一开始就加入评论、搜索、统计、订阅等功能。不是这些功能没价值，而是它们现在不是最关键的问题。

当前最关键的问题只有一个：我能不能持续写，并且让每篇内容都能自然归档到项目、工具、面经、随笔这些栏目里。

所以我更愿意先接受一个朴素但可控的系统。Astro 负责页面和构建，JSON 负责内容存储，后台负责最基本的维护入口，Nginx 和 Node 负责线上运行。等内容真的写起来，再决定哪些功能值得继续加。

## 下一阶段要做什么

云端部署完成之后，博客才真正进入内容阶段。

接下来我会优先补三类内容：第一类是项目开发日志，记录每个项目从想法、实现到复盘的过程；第二类是工具箱，把环境配置、常用脚本、部署经验整理成可复用清单；第三类是面经和履历相关内容，把求职准备过程中的问题和反思沉淀下来。

技术上，我还想继续补强几个地方：更稳妥的备份机制、更严格的后台鉴权、更舒服的文章编辑体验，以及站点地图、RSS 和搜索体验的持续完善。

但说到底，这个博客最重要的功能不是某个按钮，也不是某个框架。

它最重要的功能，是逼我把做过的事情讲清楚。只要这个习惯能坚持下去，这个站就已经开始发挥作用了。
